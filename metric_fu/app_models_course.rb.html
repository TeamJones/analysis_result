1# Course represents a course taught at CMU-SV.
  
2#
  
3# At present, there is no distinction between sections of a course. Each team can belong to a "section"
  
4# which is only a labeled text field.
  
5#
  
6# == Adding a Course Offering
  
7#
  
8# Sometime before a new semester starts, Gerry will create all the courses for the next semester. The HUB
  
 9# has a deadline for updating their system with courses, once that deadline has passed, Gerry typically will
  
10# update the rails system. A course can be added/removed/modified after this deadline, and the information in
  
11# the rails system should be updated. When Gerry creates a course, its will the minimal necessary information.
  
12# When the course is created, certain information is copied from the previous offering of the same course, where as
  
13# other information must not be copied.
  
14#
  
15# The CMU-SV community typically does not refer to courses by their number, where as on the Pittsburgh campus,
  
16# most undergraduate courses are referred to by their number.
  
17#
  
18# The system asks for the tuple (course_number, semester, and year) to create the course and then puts the user
  
19# in an edit mode prompting reasonable defaults from the last time the course was offered. If nothing has changed,
  
20# it's easy for Gerry to create the next course. If the instructor has changed then it's easy to edit that information
  
21#
  
22#
  
23# == Notifying instructors
  
24#
  
25# Whenever an instructor is added to a course, they are notified about the change, asking them to review the course
  
26# options. We ask that one of the instructors confirm the settings, when this happens we consider that the faculty
  
27# has "configured" the course. (Or verified it's settings.) If this doesn't happen, the system should periodically
  
28# remind faculty about the change.)
  
29#
  
30# Course has grading rules. These include grading cut_offs for grade's like A,A-,B+ etc.
